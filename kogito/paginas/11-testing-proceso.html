Tema 11

Probar el proceso: JUnit y Mockito

Un proceso es código que se ejecuta. Y todo código que se ejecuta merece pruebas automatizadas. Aquí pondrás bajo red de seguridad tus decisiones, tu flujo y tu endpoint REST — con las mismas herramientas que ya dominas.

Por qué probar un proceso

Un proceso de negocio cambia. El día de mañana el área de riesgo dirá «ahora aprobamos hasta 300 000 si el score es 720» o «agreguen una rama de revisión manual para montos altos». Cada uno de esos cambios mueve una condición de un gateway, ajusta una regla o reordena una tarea. Y cada cambio puede romper, sin que nadie lo note, un camino que ayer funcionaba.

Esa es exactamente la situación en la que las pruebas de regresión automatizadas ganan su sueldo: corres la batería completa, en segundos, y sabes si tu cambio de hoy rompió un comportamiento de ayer. Sin pruebas, la única red de seguridad es probar a mano cada rama en cada despliegue — algo que nadie hace de forma fiable más de un par de veces.

No estás aprendiendo un sistema de pruebas nuevo. Son los mismos JUnit 5 y Mockito del Módulo 5, las mismas assertThat(...) de AssertJ, el mismo @SpringBootTest. Lo único que cambia es qué pruebas: en lugar de un Service, ahora pruebas un proceso BPMN y una decisión DMN. La mecánica es idéntica.

ℹ️ Nota del capstone

Todos los ejemplos de esta página corresponden al proyecto del capstone (carpeta proyecto/aprobacion-credito): el proceso aprobacionCredito, el modelo SolicitudCredito y la decisión DMN que ya conoces de los temas anteriores. En el Tema 12 los juntarás todos.

La pirámide de pruebas del proceso

Como en cualquier sistema, conviene probar mucho en lo barato y rápido, y poco en lo caro y lento. La base de la pirámide son pruebas pequeñas y veloces que aíslan una pieza; la punta son pruebas amplias que ejercitan todo junto pasando por la red. Aplicado a un servicio Kogito, queda en tres niveles bien definidos.

  1. Prueba de decisión (DMN) aislada. Evalúas el modelo DMN con un conjunto de datos de entrada y compruebas la salida. No arrancas el motor de procesos ni levantas el servidor: es prácticamente una prueba unitaria. Aquí cabe la combinatoria fina (cada tramo de score, cada límite de monto), porque cada caso cuesta milisegundos.
  2. Prueba del proceso. Arrancas una instancia del proceso aprobacionCredito con una SolicitudCredito de entrada y verificas dos cosas: que la instancia llegó al estado esperado (COMPLETED) y que las variables de salida quedaron como debían. Esto valida que el flujo, los gateways y las tareas estén bien cableados.
  3. Prueba de integración REST. Levantas la aplicación completa y golpeas el endpoint autogenerado (POST /aprobacionCredito) con un JSON real. Validas el contrato HTTP de extremo a extremo: códigos de estado, forma del cuerpo, serialización. Pocas, pero imprescindibles.
✅ Tip

La regla práctica: lo que puedas probar en el nivel 1, no lo pruebes en el 3. Si una rama depende solo de la lógica de la decisión, valídala como decisión aislada (rápida) y deja para el nivel REST únicamente la verificación del contrato y la integración. Tu suite se mantendrá veloz y dará retroalimentación inmediata.

Nivel 2: probar el proceso con JUnit 5 y la API de Kogito

Kogito expone una API Java limpia para manejar procesos en pruebas. La pieza central es org.kie.kogito.process.Processes: un bean que Spring inyecta y que actúa como catálogo de todos los procesos compilados en tu aplicación. A partir de él obtienes el proceso por su ID, le pasas un modelo con la variable de entrada, creas una instancia y la arrancas.

El ciclo de una instancia en una prueba

El patrón siempre es el mismo: obtener proceso → crear modelo → crear instancia → arrancar → afirmar estado y variables. Este pequeño diagrama lo resume.

AprobacionCreditoProcessTest.java
package com.academy.kogito.proceso;

import com.academy.kogito.model.SolicitudCredito;
import org.junit.jupiter.api.Test;
import org.kie.kogito.Model;
import org.kie.kogito.process.Process;
import org.kie.kogito.process.ProcessInstance;
import org.kie.kogito.process.Processes;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.context.SpringBootTest;

import java.math.BigDecimal;
import java.util.Map;

import static org.assertj.core.api.Assertions.assertThat;

// Levanta el contexto de Spring: Kogito ya compiló el proceso
// y registró el bean Processes que vamos a inyectar.
@SpringBootTest
class AprobacionCreditoProcessTest {

    @Autowired
    Processes processes; // catálogo de todos los procesos de la app

    @Test
    void solicitudConBuenScoreTerminaAprobada() {
        // 1) Obtener el proceso por su ID (el del archivo .bpmn)
        Process<? extends Model> proceso =
                processes.processById("aprobacionCredito");

        // 2) Preparar la variable de entrada del proceso
        SolicitudCredito solicitud = new SolicitudCredito();
        solicitud.setNombre("Ana Pérez");
        solicitud.setIngresosMensuales(new BigDecimal("45000"));
        solicitud.setMontoSolicitado(new BigDecimal("120000"));
        solicitud.setPlazoMeses(24);
        solicitud.setTieneMora(false);

        // 3) Empaquetar la variable en el modelo del proceso
        Model modelo = proceso.createModel();
        modelo.fromMap(Map.of("solicitud", solicitud));

        // 4) Crear y arrancar la instancia
        ProcessInstance<? extends Model> instancia =
                proceso.createInstance(modelo);
        instancia.start();

        // 5) Afirmar el ESTADO: el proceso terminó por completo
        assertThat(instancia.status())
                .isEqualTo(ProcessInstance.STATE_COMPLETED); // == 2

        // 6) Afirmar las VARIABLES de salida
        Map<String, Object> salida = instancia.variables().toMap();
        SolicitudCredito resultado =
                (SolicitudCredito) salida.get("solicitud");

        assertThat(resultado.getDecision()).isEqualTo("APROBADO");
        assertThat(resultado.getScoreCrediticio()).isGreaterThanOrEqualTo(700);
        assertThat(resultado.getEvaluacion()).isNotNull();
    }
}
ℹ️ Sobre status() y los estados

ProcessInstance.status() devuelve un entero con los estados de Kogito: STATE_ACTIVE (1), STATE_COMPLETED (2), STATE_ABORTED (3), STATE_SUSPENDED (4) y STATE_ERROR (5). Si el proceso tuviera una tarea humana en medio, tras start() quedaría en STATE_ACTIVE esperando; ahí completarías la tarea con la API y volverías a afirmar.

Fíjate en el assertThat(...) de AssertJ y en el @Test de JUnit 5: son idénticos a los que escribías para probar un Service de Spring. El «Working with assertions» que practicaste —encadenar .isEqualTo(), .isGreaterThanOrEqualTo(), .isNotNull() para mensajes de error legibles— se aplica tal cual sobre las variables del proceso.

✅ Tip · una prueba por camino

Escribe un @Test por cada rama del proceso: uno para APROBADO, otro para REVISION, otro para RECHAZADO. Así cada cambio futuro en un gateway o una regla disparará exactamente la prueba que debe fallar, y sabrás de inmediato qué camino se rompió.

Nivel 1: probar la decisión (DMN) aislada

Antes de probar todo el flujo, conviene clavar la lógica de negocio donde vive: la decisión DMN que clasifica la solicitud. Kogito expone org.kie.kogito.decision.DecisionModels para evaluar un modelo DMN sin arrancar el motor de procesos. Le pasas un mapa de entradas, ejecuta el modelo y te devuelve el resultado.

EvaluacionCreditoDmnTest.java
package com.academy.kogito.decision;

import org.junit.jupiter.api.Test;
import org.kie.kogito.decision.DecisionModel;
import org.kie.kogito.decision.DecisionModels;
import org.kie.kogito.dmn.rest.DMNResult;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.context.SpringBootTest;

import java.util.Map;

import static org.assertj.core.api.Assertions.assertThat;

@SpringBootTest
class EvaluacionCreditoDmnTest {

    @Autowired
    DecisionModels decisionModels;

    @Test
    void scoreAltoYMontoBajoDaAprobado() {
        // Localiza el modelo por su namespace y nombre (los del .dmn)
        DecisionModel modelo = decisionModels.getDecisionModel(
                "https://academy.com/dmn/scoringCredito", "scoringCredito");

        // Entradas del DMN, igual que las columnas de tu tabla de decisión
        Map<String, Object> entradas = Map.of(
                "score", 730,
                "monto", 120000,
                "tieneMora", false);

        DMNResult resultado =
                modelo.evaluateAll(modelo.newContext(entradas));

        // El DMN no tuvo errores y la decisión es la esperada
        assertThat(resultado.hasErrors()).isFalse();
        assertThat(resultado.getDecisionResultByName("evaluacion")
                .getResult()).isEqualTo("APROBADO");
    }
}
⚠️ Ojo · namespace y nombre del DMN

El primer argumento de getDecisionModel(...) es el namespace del modelo y el segundo su nombre — ambos definidos en el archivo .dmn. Si los confundes, obtendrás un null y un NullPointerException al evaluar. Cópialos tal cual del editor de DMN.

Como cada evaluación cuesta microsegundos, este es el lugar para la combinatoria exhaustiva. Una prueba parametrizada de JUnit 5 cubre todos los tramos de score de un solo método.

EvaluacionCreditoDmnTest.java (parametrizada)
    @ParameterizedTest
    @CsvSource({
        "800, 100000, false, APROBADO",
        "650, 100000, false, REVISION",
        "500, 100000, false, RECHAZADO",
        "800, 100000, true,  RECHAZADO"   // con mora: siempre rechazado
    })
    void clasificaSegunScoreYMora(int score, int monto,
                                boolean tieneMora, String esperado) {
        DecisionModel modelo = decisionModels.getDecisionModel(
                "https://academy.com/dmn/scoringCredito", "scoringCredito");

        DMNResult r = modelo.evaluateAll(modelo.newContext(Map.of(
                "score", score, "monto", monto, "tieneMora", tieneMora)));

        assertThat(r.getDecisionResultByName("evaluacion").getResult())
                .isEqualTo(esperado);
    }

Mockito para aislar lo que rodea al proceso

Tu proceso no vive solo: algunas tareas llaman a colaboradores externos. Por ejemplo, una tarea de servicio que, al final del flujo, invoca una pasarela de notificación para avisar al solicitante por correo o SMS. En una prueba no quieres mandar correos de verdad: quieres aislar el proceso y verificar que habría notificado, sin tocar el mundo exterior.

Esto es Mockito puro, el del Módulo 5. Reemplazas el colaborador real por un mock con @MockBean, programas su comportamiento con when(...) y, tras arrancar el proceso, compruebas con verify(...) que el proceso lo llamó como debía.

NotificacionProcessTest.java
package com.academy.kogito.proceso;

import com.academy.kogito.model.SolicitudCredito;
import com.academy.kogito.servicio.PasarelaNotificacion;
import org.junit.jupiter.api.Test;
import org.kie.kogito.Model;
import org.kie.kogito.process.Process;
import org.kie.kogito.process.ProcessInstance;
import org.kie.kogito.process.Processes;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.boot.test.mock.mockito.MockBean;

import java.math.BigDecimal;
import java.util.Map;

import static org.mockito.ArgumentMatchers.any;
import static org.mockito.ArgumentMatchers.contains;
import static org.mockito.Mockito.verify;
import static org.mockito.Mockito.when;
import static org.assertj.core.api.Assertions.assertThat;

@SpringBootTest
class NotificacionProcessTest {

    @Autowired
    Processes processes;

    // Sustituye el bean real de notificación por un mock en el contexto
    @MockBean
    PasarelaNotificacion pasarela;

    @Test
    void alAprobarSeNotificaAlSolicitante() {
        // Programamos el mock: cuando le pidan notificar, finge éxito
        when(pasarela.enviar(any(), any())).thenReturn(true);

        SolicitudCredito solicitud = new SolicitudCredito();
        solicitud.setNombre("Luis Ramírez");
        solicitud.setIngresosMensuales(new BigDecimal("60000"));
        solicitud.setMontoSolicitado(new BigDecimal("100000"));
        solicitud.setTieneMora(false);

        Process<? extends Model> proceso =
                processes.processById("aprobacionCredito");
        Model modelo = proceso.createModel();
        modelo.fromMap(Map.of("solicitud", solicitud));

        ProcessInstance<? extends Model> instancia =
                proceso.createInstance(modelo);
        instancia.start();

        assertThat(instancia.status())
                .isEqualTo(ProcessInstance.STATE_COMPLETED);

        // Verificamos que el proceso SÍ llamó a la pasarela,
        // exactamente una vez y con un mensaje de aprobación.
        verify(pasarela).enviar(contains("Luis Ramírez"),
                                contains("APROBADO"));
    }
}
✅ Tip · verify cuenta llamadas

Por defecto verify(pasarela) exige exactamente una llamada. Para afirmar que un camino no notifica —por ejemplo, una rama de rechazo silencioso— usa verify(pasarela, never()).enviar(any(), any()). Igual que en tus pruebas de Service.

⛔ Error común

No confundas mockear el colaborador con mockear el proceso. El proceso es lo que estás probando: déjalo correr de verdad con la API de Kogito. El mock es solo para las dependencias alrededor (correo, pagos, servicios remotos). Si mockeas el proceso, no pruebas nada.

Esto es, literalmente, el when(...).thenReturn(...) y el verify(...) del Módulo 5. La única novedad es @MockBean: en vez de inyectar el mock a mano en un constructor, le dices a Spring que lo coloque en el contexto, para que la tarea de servicio del proceso reciba el sustituto en lugar del bean real.

Nivel 3: prueba de integración REST del endpoint

En la punta de la pirámide validas el contrato HTTP completo. Recuerda: el endpoint del proceso se autogenera a partir del ID aprobacionCredito — tú nunca escribes un @RestController. Para probarlo levantas la app en un puerto real y la golpeas con REST Assured (o con TestRestTemplate), exactamente como probarías cualquier API de Spring Boot.

AprobacionCreditoRestIT.java
package com.academy.kogito.rest;

import io.restassured.http.ContentType;
import org.junit.jupiter.api.Test;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.boot.test.web.server.LocalServerPort;

import static io.restassured.RestAssured.given;
import static org.hamcrest.Matchers.equalTo;

// Levanta el servidor en un puerto aleatorio: integración real.
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class AprobacionCreditoRestIT {

    @LocalServerPort
    int puerto;

    @Test
    void postCreaInstanciaYDevuelveDecision() {
        given()
            .port(puerto)
            .contentType(ContentType.JSON)
            .body("""
                {
                  "solicitud": {
                    "nombre": "Ana Pérez",
                    "ingresosMensuales": 45000,
                    "montoSolicitado": 120000,
                    "plazoMeses": 24,
                    "tieneMora": false
                  }
                }
                """)
        .when()
            // Endpoint AUTOGENERADO a partir del id del proceso
            .post("/aprobacionCredito")
        .then()
            .statusCode(201)               // Created: la instancia se creó
            .body("solicitud.decision", equalTo("APROBADO"))
            .body("id", org.hamcrest.Matchers.notNullValue());
    }
}
ℹ️ Nota · sufijo IT

Por convención de Maven, las pruebas de integración terminan en ...IT y las corre el plugin failsafe en la fase verify, no surefire en test. Así separas la suite rápida (niveles 1 y 2) de la lenta (nivel 3) y puedes correr solo la primera durante el desarrollo.

Resumen: qué prueba cada nivel y con qué

Nivel de pruebaQué validaHerramientas
1 · Decisión (DMN) La lógica de negocio aislada: dado un score, monto y tieneMora, la decisión evaluacion es la correcta. Sin arrancar el motor de procesos. JUnit 5 · AssertJ · DecisionModels / DecisionModel
2 · Proceso El flujo completo: arranca la instancia con una SolicitudCredito, recorre los gateways y tareas, termina en COMPLETED y deja las variables esperadas. JUnit 5 · AssertJ · Processes / Process / ProcessInstance · Mockito (@MockBean) para aislar colaboradores
3 · REST El contrato HTTP de extremo a extremo del endpoint autogenerado: código de estado, forma del JSON, serialización. Integración real con el servidor levantado. JUnit 5 · @SpringBootTest(RANDOM_PORT) · REST Assured / TestRestTemplate · Maven failsafe
● Mismos JUnit y Mockito del Módulo 5 ● Decisión rápida en la base ● Proceso en el centro ● REST en la punta

El cierre del tema es el mismo mensaje con el que empezó: no hay herramientas nuevas que aprender. Son los mismos JUnit y Mockito que ya usas a diario, aplicados ahora al proceso y a la decisión. Lo único distinto son las tres APIs de Kogito que actúan como punto de entrada —Processes, DecisionModels y el endpoint REST autogenerado— y, a partir de ahí, todo es assertThat, when y verify de toda la vida.
