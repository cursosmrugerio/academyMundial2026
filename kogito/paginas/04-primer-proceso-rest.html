Tema 04

Tu primer proceso y su API REST

Dibujas un proceso de tres piezas, le pones un identificador… y Kogito te entrega, gratis, una API REST completa, versionada por proceso y lista para invocarse con curl.

Anatomía de un archivo .bpmn2

Un proceso de negocio en Kogito vive en un archivo con extensión .bpmn2 (o .bpmn). Por dentro es XML conforme al estándar BPMN 2.0, pero casi nunca lo escribes a mano: lo editas visualmente arrastrando figuras en el editor de procesos (el plugin de VS Code o el canvas de Kogito). El editor pinta el diagrama y, en paralelo, mantiene el XML sincronizado.

Ese archivo se coloca junto al código, normalmente en src/main/resources/. Cuando compilas con Maven, el procesador de Kogito lo lee en tiempo de compilación y genera el código Java —incluida la capa REST— que terminará dentro de tu binario de Spring Boot.

Piensa en el .bpmn2 como tu schema.sql o una entidad @Entity de JPA: un recurso declarativo que vive en resources y del que, al compilar, nace código real. Tú describes el qué; el generador produce el cómo.

La pieza clave: el processId

Dentro del XML, el elemento <process> lleva un atributo id: el processId. Es el nombre técnico, único, del proceso. Recuérdalo bien porque marcará toda la lección:

  • Identifica al proceso de forma única dentro de la aplicación.
  • Se convierte literalmente en la ruta REST que Kogito autogenera: /{processId}.
  • Aparece en los eventos, las métricas y las consultas del Data Index.
⚠️ Ojo

No confundas tres nombres distintos: el nombre del archivo (saludo.bpmn2), el id del proceso (el processId, que será la ruta) y el name (etiqueta legible para humanos). La ruta sale del id, no del nombre del archivo. Usa identificadores sin espacios ni acentos: saludo, no Mi Saludo.

Un proceso mínimo: saludo

El proceso más pequeño que vale la pena ejecutar tiene tres elementos: un evento de inicio, una tarea y un evento de fin, unidos por flujos de secuencia. Vamos a llamarlo saludo: recibe un nombre, arma un mensaje de bienvenida y termina.

Evento de inicio Tarea (script) Evento de fin Flujo de secuencia

Así se ve el corazón del XML que el editor mantiene por ti. No necesitas memorizarlo —fíjate solo en el id="saludo" del <process>:

src/main/resources/saludo.bpmn2
<!-- Fragmento ilustrativo y recortado: el editor visual genera el resto -->
<process id="saludo" name="Saludo de bienvenida"
         processType="Public" isExecutable="true">

  <!-- Variable de proceso: el contrato de datos del endpoint -->
  <property id="nombre"  itemSubjectRef="_nombreItem"/>
  <property id="saludo"  itemSubjectRef="_saludoItem"/>

  <startEvent   id="_start" name="Inicio"/>

  <scriptTask  id="_saludar" name="Construir saludo">
    <script>kcontext.setVariable("saludo",
        "¡Hola, " + nombre + "! Bienvenido a Kogito.");</script>
  </scriptTask>

  <endEvent     id="_end" name="Fin"/>

  <!-- Flujos de secuencia que conectan los nodos -->
  <sequenceFlow sourceRef="_start"   targetRef="_saludar"/>
  <sequenceFlow sourceRef="_saludar" targetRef="_end"/>
</process>
✅ Tip

Las dos <property> —nombre y saludo— son variables del proceso. Aquí está la magia: esas variables forman el contrato JSON de tu API. Lo que mandas al arrancar (nombre) y lo que recibes de vuelta (saludo) salen directamente de ellas. Profundizaremos en variables y tipos en el próximo tema.

De processId a API REST, sin escribir un controlador

Aquí viene lo que distingue a Kogito. En cuanto el proyecto compila, el generador toma tu proceso saludo y produce un conjunto de endpoints REST específicos de ese proceso. No hay un @RestController en tu código: el controlador se genera y se compila dentro del binario. La ruta base es el processId: /saludo.

Cada endpoint opera sobre instancias del proceso. Una instancia es una ejecución concreta —como una fila en una tabla— y se identifica con un id que Kogito genera (un UUID):

  • POST /saludo — arranca una nueva instancia. Le envías los datos de entrada en el cuerpo JSON y te devuelve el id de la instancia recién creada junto con sus variables.
  • GET /saludo — lista las instancias activas de ese proceso.
  • GET /saludo/{id} — consulta una instancia concreta por su id.
  • DELETE /saludo/{id} — aborta (cancela) esa instancia.
ℹ️ Nota

Nuestro proceso saludo es directo: arranca y termina en el mismo POST porque no tiene esperas (ni tareas humanas ni temporizadores). Por eso, al crearlo, ya recibes el resultado calculado. En procesos que se quedan a la espera, la instancia sigue viva y la consultas después con GET /saludo/{id}.

Conexión con el Módulo 6: verbos y códigos de estado

Todo lo que aprendiste sobre HTTP aplica tal cual. Kogito respeta la semántica REST: el verbo indica la intención y el código de respuesta indica el resultado. Esta es la tabla de endpoints autogenerados para saludo:

VerboRutaQué haceCódigo típico
POST /saludo Crea (arranca) una instancia del proceso con los datos de entrada 201 Created
GET /saludo Lista las instancias activas 200 OK
GET /saludo/{id} Consulta una instancia concreta 200 OK · 404 Not Found
DELETE /saludo/{id} Aborta (cancela) la instancia 200 OK · 404 Not Found
cualquiera /saludo… Cuerpo mal formado o validación fallida 400 Bad Request

Es exactamente el contrato REST del Módulo 6: POST que crea responde 201 Created (a menudo con cabecera Location apuntando a la nueva instancia); GET que encuentra responde 200 OK; pedir un id inexistente da 404 Not Found; y un cuerpo inválido da 400 Bad Request. Lo que cambia es que no escribiste el controlador: lo generó el proceso.

⚠️ Ojo

Verás documentación que muestra POST respondiendo 200 OK en lugar de 201 Created según la versión y la configuración. La semántica del Módulo 6 es la guía: una creación exitosa debería ser 201. No te ancles al número exacto: razona por la intención del verbo.

Manos a la obra: invócalo con curl

Con la aplicación corriendo (mvn spring-boot:run, por defecto en el puerto 8080), arrancar una instancia es un simple POST. El cuerpo JSON lleva las variables de entrada del proceso —en nuestro caso, nombre:

terminal · arrancar una instancia
# POST a /saludo con la variable de entrada en el cuerpo
curl -i -X POST http://localhost:8080/saludo \
  -H "Content-Type: application/json" \
  -d '{ "nombre": "Ana" }'

La respuesta trae el id de la instancia y el estado completo de sus variables, incluido el saludo ya calculado:

respuesta HTTP
HTTP/1.1 201 Created
Location: /saludo/2f4a1c9e-7b3d-4a16-9f0e-8c2d11ab77e1
Content-Type: application/json

{
  "id": "2f4a1c9e-7b3d-4a16-9f0e-8c2d11ab77e1",
  "nombre": "Ana",
  "saludo": "¡Hola, Ana! Bienvenido a Kogito."
}

Ese id es la llave para el resto de operaciones. Consulta o aborta la instancia con él:

terminal · consultar y abortar
# Listar instancias activas
curl http://localhost:8080/saludo

# Consultar una instancia por su id
curl http://localhost:8080/saludo/2f4a1c9e-7b3d-4a16-9f0e-8c2d11ab77e1

# Abortar (cancelar) esa instancia
curl -X DELETE http://localhost:8080/saludo/2f4a1c9e-7b3d-4a16-9f0e-8c2d11ab77e1
✅ Tip

Como todo endpoint REST autogenerado expone su contrato, Kogito también publica un OpenAPI en /q/openapi y un Swagger UI para explorar y probar las rutas desde el navegador. Tus pruebas con MockMvc o RestAssured de JUnit funcionan igual que contra cualquier controlador de Spring Boot.

El ciclo completo, de un vistazo

Recapitulando el camino desde el archivo hasta la llamada REST:

  1. Dibujas Inicio → Tarea → Fin en el editor visual; se guarda como saludo.bpmn2 en resources.
  2. Le das al proceso un processId: saludo.
  3. Al compilar, Kogito genera el controlador REST con base /saludo y sus operaciones.
  4. Levantas la app con spring-boot:run y haces POST /saludo.
  5. Recibes 201 Created con el id de instancia y las variables de salida.

Detente en lo logrado: no escribiste ni un @RestController, ni un @PostMapping, ni un DTO, y ya tienes una API REST versionada por proceso —cada proceso es su propio recurso bajo su propia ruta— cuyo contrato JSON sale de las variables del proceso. En Spring partías del controlador hacia el dominio; aquí partes del dominio (el proceso) y el controlador cae solo. En el siguiente tema veremos cómo declarar esas variables, tiparlas y modelar el JSON de entrada y salida.

⛔ Importante

Error común: tratar de “añadir” un @RestController propio para exponer el proceso. No hace falta y se pisaría con el generado. Si necesitas una ruta a la medida, se hace con la API programática de Kogito (ProcessService y compañía), no replicando el controlador autogenerado a mano.
