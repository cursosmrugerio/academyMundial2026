Proyecto final

Capstone: proceso de aprobación de crédito

Es la hora de juntar todas las piezas. Vamos a construir un proceso bancario real, ejecutable de punta a punta: una solicitud entra por REST, unas reglas calculan el score, una compuerta decide el destino y, cuando hace falta, un analista humano da el visto bueno. Todo lo que viste en el curso, vivo en un solo proyecto Maven.

El caso: «Crédito Express» del banco

Imagina la mesa de crédito de un banco. Un cliente solicita un préstamo desde la banca en línea. El proceso debe, en segundos, decidir entre tres desenlaces: APROBADO (desembolso inmediato), RECHAZADO (no califica) o REVISION (un caso de frontera que un analista de carne y hueso debe mirar antes de resolver).

Eso es exactamente lo que vamos a modelar. El proyecto se llama aprobacion-credito, el proceso tiene processId = aprobacionCredito y su endpoint —que, como ya sabes, se autogenera— responde en POST /aprobacionCredito. La lógica de puntaje vive en una decisión DMN llamada scoringCredito. Las variables del proceso son dos: solicitud (lo que manda el cliente) y evaluacion (lo que produce la decisión).

Si esto fuera un microservicio de Spring puro, tendrías un @RestController con la solicitud, un @Service con los if del puntaje, una cola o tabla para los casos en revisión y un controlador extra para que el analista los resuelva. Aquí, el diagrama del proceso es el orquestador: el endpoint, el enrutamiento, la tarea humana y la espera los pone Kogito. Tú aportas el modelo de datos, las reglas y, si quieres, algún servicio de apoyo.

Inicio / Fin = eventos Reglas y tarea humana = tareas Decisión = gateway exclusivo Flujo de secuencia

El proceso completo, de izquierda a derecha

Este es el corazón del capstone: el flujo de punta a punta. Léelo como una historia. Entra una solicitud por la izquierda; una tarea de reglas la evalúa con la decisión DMN; un gateway exclusivo lee evaluacion.decision y abre tres caminos; dos terminan de inmediato y el tercero pasa por un analista humano antes de cerrar.

Cada pieza y el tema del curso que le corresponde

Lo bonito del capstone es que cada figura del diagrama es un tema que ya estudiaste. Esta tabla es tu mapa: pieza por pieza, dónde lo aprendiste.

Pieza del diagramaElemento BPMNTema del cursoQué hace aquí
Inicio Evento de inicio (none) Eventos · Primer proceso por REST Recibe el POST y crea la instancia con la variable solicitud ya cargada.
Evaluar score Business Rule Task Business Rule Task / DMN Invoca la decisión scoringCredito, calcula el score y deja el resultado en evaluacion.
¿decision? Gateway exclusivo (XOR) Gateways y decisiones Lee evaluacion.decision y enruta a una sola de las tres ramas.
Revisión del analista User Task (grupo analistas) Tareas humanas Pausa el proceso y espera la resolución de una persona; expone endpoints de tarea autogenerados.
solicitud / evaluacion Variables del proceso Variables y datos solicitud es la entrada (tag input); evaluacion es la salida (tag output).
Fin aprobado / rechazado / revisado Eventos de fin Eventos Cierran la instancia. La respuesta del REST trae el estado final del proceso y sus variables.
✅ Tip

Fíjate en la separación de responsabilidades: el «qué decidir» (el puntaje, el umbral) vive en el DMN, y el «por dónde seguir» vive en el gateway. Mantenerlos separados hace que un cambio de política de riesgo se haga tocando solo la tabla de decisión, sin redibujar el proceso. Es el mismo principio de no mezclar reglas de negocio con enrutamiento que ya aplicas en Spring.

Estructura del proyecto Maven

El proyecto es un Spring Boot normal, con un detalle: dentro de src/main/resources conviven los recursos de Kogito (el .bpmn2 y el .dmn) junto al application.properties de siempre. En tiempo de compilación, Kogito lee esos archivos y genera el endpoint y el cableado. Tú no escribes ningún controlador.

ℹ️ Nota

La extensión .bpmn2 y .dmn es lo que Kogito busca en el classpath al compilar. El nombre del archivo no fija el processId: el processId es un atributo dentro del XML del proceso (aquí, aprobacionCredito) y es ese identificador el que define la ruta del endpoint, no el nombre del archivo.

El modelo de datos: lo que ya sabes de POJOs y JPA

Las variables del proceso son POJOs de Java. SolicitudCredito es lo que el cliente manda; Evaluacion es lo que la decisión produce y lo que el gateway consulta. Son clases planas, sin magia.

model/SolicitudCredito.java
package com.academy.kogito.model;

public class SolicitudCredito {
    private String nombre;
    private int edad;
    private BigDecimal ingresosMensuales;
    private BigDecimal montoSolicitado;
    private int plazoMeses;
    private boolean tieneMora;

    // getters y setters — Kogito (de)serializa con estos
}
model/Evaluacion.java
package com.academy.kogito.model;

public class Evaluacion {
    private int score;          // puntaje calculado por el DMN
    private String decision;     // "APROBADO" | "RECHAZADO" | "REVISION"
    private String motivo;       // texto explicativo

    // getters y setters
}

Estos POJOs son tus @Entity/DTO de toda la vida, pero sin anotaciones de persistencia obligatorias para que el proceso funcione. El estado del proceso se guarda como variables; si más adelante activas la persistencia JDBC, esas variables se serializan a PostgreSQL sin que cambies el POJO. Lo mismo que cuando pasas un DTO a JSON: el motor hace el (de)serialize por ti.

La decisión DMN: scoringCredito

El DMN recibe la solicitud como input data y produce una decisión final con el puntaje y el veredicto. En el editor lo dibujas como una tabla de decisión con política de aciertos First (gana la primera fila que coincida). Conceptualmente, la lógica es esta:

scoringCredito — tabla de decisión (FEEL)
// Decisión "Evaluacion" · Hit policy: First (F)
// Inputs (de la solicitud):  edad, ingresosMensuales, montoSolicitado, tieneMora

// Regla 1 — descarte por mora
tieneMora = true                                  →  { score: 350,  decision: "RECHAZADO", motivo: "Mora vigente" }

// Regla 2 — capacidad de pago holgada → aprobado directo
montoSolicitado <= ingresosMensuales * 6          →  { score: 760,  decision: "APROBADO",  motivo: "Capacidad suficiente" }

// Regla 3 — caso de frontera → lo ve un analista
montoSolicitado <= ingresosMensuales * 12         →  { score: 650,  decision: "REVISION",  motivo: "Monto alto: revisar" }

// Regla por defecto — sobreendeudamiento
otherwise                                         →  { score: 480,  decision: "RECHAZADO", motivo: "Monto excede capacidad" }
⚠️ Ojo

En FEEL los nombres con espacios son válidos (ingresos mensuales), pero para que el binding con tu POJO sea limpio, nombra los input data del DMN igual que las propiedades de SolicitudCredito. Aquí mantenemos ingresosMensuales, montoSolicitado y tieneMora idénticos a los getters del POJO.

«Evaluar score»: el cálculo de la decisión

En este proyecto, la tarea Evaluar score es una Script Task que calcula el score y la decision en Java a partir de solicitud y deja el resultado en la variable evaluacion. Se hace así para que el ejemplo corra sin configuración extra y de forma determinista. La misma lógica de negocio está modelada como decisión DMN (scoringCredito.dmn), desplegada como servicio independiente (POST /scoringCredito) y probada de forma aislada en ScoringDmnTest.

aprobacionCredito.bpmn2 — Script Task «Evaluar score» (fragmento)
int score = puntosIngresos + puntosRatio + puntosEdad + puntosMora;
String decision = score >= 70 ? "APROBADO"
               : score >= 50 ? "REVISION"
               : "RECHAZADO";

Evaluacion e = new Evaluacion();
e.setScore(score); e.setDecision(decision); e.setMotivo(motivo);
kcontext.setVariable("evaluacion", e);

En un despliegue productivo, «Evaluar score» sería una Business Rule Task que invoca el DMN scoringCredito (por su namespace https://academy.com/dmn/scoringCredito y su nombre scoringCredito) en lugar de calcular dentro del proceso. Es la misma decisión, externalizada y editable por el área de negocio — justo lo del Tema 08.

Y la condición de cada salida del gateway exclusivo, sobre evaluacion.decision:

Condiciones del gateway (dialecto Java)
// Flujo → Fin aprobado
"APROBADO".equals(evaluacion.getDecision())

// Flujo → Revisión del analista (User Task)
"REVISION".equals(evaluacion.getDecision())

// Flujo → Fin rechazado   ➜  marcado como "Por defecto", sin expresión (tu else)

Cómo ejecutarlo, paso a paso

Con el proyecto en tu máquina y un JDK 21 instalado, el ciclo es idéntico al de cualquier Spring Boot. La diferencia es que el endpoint ya está listo sin que escribas un controlador.

  1. Levanta la aplicación. Desde la carpeta del proyecto, modo desarrollo de Spring Boot:
    terminal
    cd proyecto/aprobacion-credito
mvn spring-boot:run

    En el arranque, Kogito compila aprobacionCredito.bpmn2 y scoringCredito.dmn y publica el endpoint POST /aprobacionCredito.

  2. Arranca un proceso con un POST. El cuerpo es el JSON de la SolicitudCredito envuelto en la variable solicitud:
    curl — solicitud que cae en REVISION
    curl -i -X POST http://localhost:8080/aprobacionCredito \
  -H "Content-Type: application/json" \
  -d '{
    "solicitud": {
      "nombre": "Ana López",
      "edad": 34,
      "ingresosMensuales": 25000,
      "montoSolicitado": 220000,
      "plazoMeses": 36,
      "tieneMora": false
    }
  }'
  3. Observa la decisión. Según el monto frente a los ingresos, el desenlace cambia. Con 220000 contra 25000 de ingreso (entre 6× y 12×), cae en REVISION y el proceso queda activo esperando al analista. La respuesta trae el id de la instancia y las variables:
    respuesta · HTTP 201 Created
    HTTP/1.1 201
Content-Type: application/json

{
  "id": "a1b2c3d4-0001-4f10-9a77-2c0e9f5b1234",
  "solicitud": { "nombre": "Ana López", "montoSolicitado": 220000, ... },
  "evaluacion": { "score": 650, "decision": "REVISION", "motivo": "Monto alto: revisar" }
}

    Si en cambio mandas un monto pequeño (por ejemplo 120000), la decisión es APROBADO, no hay tarea humana y el proceso termina al instante: la respuesta es HTTP 201 con el estado final. Si mandas tieneMora: true, es RECHAZADO de inmediato.

  4. Consulta y completa la tarea por REST. Para los casos en REVISION, los endpoints de tarea humana también se autogeneran. Primero listas la tarea pendiente de la instancia:
    curl — listar tareas de la instancia
    curl http://localhost:8080/aprobacionCredito/{id}/tasks?user=marta&group=analistas

    Eso devuelve el taskId de la tarea RevisionAnalista. La tomas y la completas con la decisión del analista (aprobar o rechazar):

    curl — completar la tarea
    # Tomar la tarea (claim)
curl -X POST "http://localhost:8080/aprobacionCredito/{id}/RevisionAnalista/{taskId}/phases/claim?user=marta&group=analistas"

# Completar con el veredicto humano
curl -X POST "http://localhost:8080/aprobacionCredito/{id}/RevisionAnalista/{taskId}?user=marta&group=analistas" \
  -H "Content-Type: application/json" \
  -d '{ "aprobadoPorAnalista": true, "comentario": "Cliente con buen historial" }'

    Al completarse la tarea, el proceso avanza a su fin de revisión y la instancia se cierra. Si no había tarea (caso APROBADO o RECHAZADO directo), este paso simplemente no aplica.

  5. Corre las pruebas. El proyecto incluye pruebas que cubren el flujo completo, la decisión aislada y el servicio de apoyo:
    terminal
    mvn test
⛔ Importante

El cuerpo del POST usa la variable del proceso como clave de primer nivel: { "solicitud": { ... } }, no los campos sueltos en la raíz. Si mandas { "nombre": ..., "monto": ... } directamente, Kogito no encontrará la variable solicitud y la evaluación trabajará con datos nulos. Mismo cuidado que con el binding del @RequestBody: la forma del JSON debe calzar con la estructura esperada.

Las pruebas: JUnit y Mockito, como siempre

Probar el proceso no requiere nada exótico. Inyectas el proceso generado, arrancas una instancia con datos y verificas el estado y las variables. Para la decisión, evalúas el modelo DMN directamente. Y para tu servicio de apoyo, JUnit + Mockito de toda la vida.

test/AprobacionCreditoProcessTest.java
@SpringBootTest
class AprobacionCreditoProcessTest {

    @Autowired // el proceso se genera al compilar y se publica como bean
    Process<? extends Model> aprobacionCredito;

    @Test
    void aprobadoCuandoElMontoEsHolgado() {
        var solicitud = new SolicitudCredito(
            "Ana López", 34, bd("25000"), bd("120000"), 36, false);

        var model = aprobacionCredito.createModel();
        model.fromMap(Map.of("solicitud", solicitud));

        var instancia = aprobacionCredito.createInstance(model);
        instancia.start();

        // Sin tarea humana → el proceso termina solo
        assertEquals(ProcessInstance.STATE_COMPLETED, instancia.status());
        var evaluacion = (Evaluacion) instancia.variables().toMap().get("evaluacion");
        assertEquals("APROBADO", evaluacion.getDecision());
    }

    @Test
    void montoFronteraQuedaEnRevisionConTareaActiva() {
        var solicitud = new SolicitudCredito(
            "Ana López", 34, bd("25000"), bd("220000"), 36, false);
        var model = aprobacionCredito.createModel();
        model.fromMap(Map.of("solicitud", solicitud));

        var instancia = aprobacionCredito.createInstance(model);
        instancia.start();

        // Hay una tarea humana pendiente → el proceso sigue activo
        assertEquals(ProcessInstance.STATE_ACTIVE, instancia.status());
        assertFalse(instancia.workItems().isEmpty());
    }
}
test/DesembolsoServiceTest.java — JUnit + Mockito
@ExtendWith(MockitoExtension.class)
class DesembolsoServiceTest {

    @Mock  CuentaRepository cuentas;     // el repositorio JPA, simulado
    @InjectMocks DesembolsoService servicio;

    @Test
    void registraElDesembolso() {
        when(cuentas.save(any())).thenAnswer(inv -> inv.getArgument(0));

        servicio.desembolsar("Ana López", bd("120000"));

        verify(cuentas).save(argThat(c -> c.getMonto().equals(bd("120000"))));
    }
}

¿Ves la continuidad? @SpringBootTest, @Mock, @InjectMocks, verify, argThat… son exactamente las herramientas que ya dominas. El proceso es un bean más que se autoconfigura; el repositorio JPA se mockea igual que en cualquier servicio. Probar un proceso de negocio no es una disciplina nueva: es tu testing de Spring aplicado a un orquestador.

Lo que lograste y hacia dónde seguir

✅ Checklist de lo aprendido

Si completaste el capstone, ya sabes:

  • Modelar un proceso de negocio real en BPMN y dejar que Kogito genere su endpoint REST sin escribir controladores.
  • Externalizar la lógica de negocio en una decisión DMN e invocarla desde una Business Rule Task.
  • Enrutar el flujo con un gateway exclusivo a partir de una variable, con su rama por defecto.
  • Incorporar una tarea humana asignada a un grupo y resolverla por REST (claim → complete).
  • Manejar variables del proceso como POJOs de entrada y salida.
  • Probar todo con JUnit y Mockito, tal como ya lo hacías en Spring.

Siguientes pasos para llevarlo a producción

El proyecto corre con estado en memoria, perfecto para aprender. Para producción, estos son los tres saltos naturales, cada uno casi sin tocar tu modelo:

  1. Persistencia en PostgreSQL. Agrega el addon jbpm-addons-springboot-persistence-jdbc y configura el datasource en application.properties (spring.datasource.url=jdbc:postgresql://...). El estado de cada instancia y cada tarea sobrevive a un reinicio. Es el mismo PostgreSQL al que ya conectas tu JPA.
  2. Eventos con Kafka. Con el addon kie-addons-springboot-events-process-kafka, Kogito publica eventos CloudEvents (instancia iniciada, tarea creada, proceso completado) que otros servicios pueden consumir. Útil para auditoría, notificaciones o disparar otros procesos.
  3. Desplegar en contenedor. Empaqueta con mvn package, construye una imagen y despliégala en Kubernetes/OpenShift. Como es un microservicio por proceso, escalas esta capacidad de crédito de forma independiente del resto del banco.
ℹ️ Nota

Observa la promesa cumplida de Kogito: pasar de «en memoria» a «PostgreSQL + Kafka + contenedor» se hace añadiendo dependencias y configuración, no reescribiendo el proceso. El diagrama, las reglas y tus POJOs no cambian. Esa es la separación entre qué hace tu negocio y cómo se opera en infraestructura.

Cierre del curso: empezaste preguntándote qué tenía de distinto un proceso de negocio frente a un servicio de Spring. La respuesta, después de este capstone, es clara: el flujo, las decisiones y la coordinación humana dejan de ser código disperso y pasan a ser artefactos declarativos —BPMN y DMN— que Kogito ejecuta y expone por REST. Tú sigues escribiendo Java, POJOs y pruebas como siempre; solo que ahora la orquestación es explícita, visible y mantenible. Enhorabuena: ya construyes procesos de negocio cloud-native.
