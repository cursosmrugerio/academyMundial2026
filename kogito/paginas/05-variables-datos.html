Tema 05

Variables y datos del proceso

El proceso no solo describe pasos: también carga consigo un estado que viaja de principio a fin. Aquí aprenderás a definir ese estado, a darle forma con tu propio modelo de dominio y a entender cómo cada tarea lee y escribe sobre él.

Las variables del proceso: el estado que viaja

Cuando arrancas un proceso, el motor crea una instancia de ese proceso. Esa instancia necesita recordar cosas mientras avanza: quién hizo la solicitud, cuánto pidió, si ya pasó la validación, cuál fue el resultado. A todo ese conjunto de datos lo llamamos variables del proceso (en inglés, process variables).

La idea clave: las variables del proceso son el estado compartido de la instancia. Se declaran una vez a nivel del proceso y permanecen vivas desde el evento de inicio hasta el evento de fin. Cada tarea puede consultarlas y modificarlas, y el motor se encarga de que ese estado sobreviva incluso si la instancia espera horas o días en una tarea humana.

Imagina un SolicitudCredito que vas pasando como parámetro de un método de Service a otro: validar(solicitud), calcularScore(solicitud), decidir(solicitud). En Spring tú eres responsable de pasar ese objeto de mano en mano y de mantenerlo en memoria.

En Kogito, ese objeto se declara como variable del proceso. El motor lo persiste por ti, lo mapea automáticamente entre tareas y lo conserva aunque el hilo de ejecución termine y se retome más tarde. Tú solo dices qué lee y qué escribe cada actividad.

ℹ️ Nota

Cada instancia tiene su propia copia de las variables. Si arrancas dos solicitudes de crédito al mismo tiempo, cada una lleva su propio solicitud sin pisarse: son dos instancias independientes con su propio estado.

Tipos de variables: primitivos y objetos de dominio

Una variable del proceso puede ser de dos grandes familias:

Tipos primitivos y envoltorios

Para datos sueltos y sencillos te apoyas en los tipos básicos de Java:

String int boolean double Integer BigDecimal

Útiles para banderas (aprobado), totales (scoreCrediticio) o mensajes (motivoRechazo).

Objetos Java / POJO

Para representar conceptos del negocio usas tus propias clases: un Plain Old Java Object con sus atributos, igual que los @Entity o los DTO que ya escribes.

Aquí entra el modelo central de nuestro proyecto: SolicitudCredito. Una sola variable de tipo objeto agrupa muchos datos relacionados y los hace viajar juntos por el flujo.

✅ Tip

Prefiere un objeto de dominio bien modelado sobre una docena de variables primitivas sueltas. Una variable solicitud de tipo SolicitudCredito es más limpia, más fácil de mapear y reaprovecha la validación y la lógica que ya tienes en tus clases.

El modelo de dominio: SolicitudCredito

Este POJO será el corazón de los datos que viajan por nuestro proceso de crédito. Es un objeto Java normal y corriente: sin nada propio de Kogito en él. Lo declaramos en el paquete del modelo de tu aplicación Spring Boot.

SolicitudCredito.java
package com.academy.credito.model;

import java.math.BigDecimal;

// POJO de dominio: el "estado" que viajará por el proceso.
// No tiene nada específico de Kogito; es Java puro y serializable.
public class SolicitudCredito {

    private String nombre;
    private int edad;
    private BigDecimal ingresosMensuales;
    private BigDecimal montoSolicitado;
    private int plazoMeses;
    private boolean tieneMora;

    // Campos que algunas tareas escribirán durante el flujo:
    private int scoreCrediticio;
    private Boolean aprobado;
    private String motivoDecision;

    public SolicitudCredito() { } // constructor vacío: lo necesita la deserialización

    // getters y setters de cada atributo…
    public String getNombre() { return nombre; }
    public void setNombre(String nombre) { this.nombre = nombre; }

    public BigDecimal getMontoSolicitado() { return montoSolicitado; }
    public void setMontoSolicitado(BigDecimal m) { this.montoSolicitado = m; }

    // … (resto de getters/setters) …
}
⚠️ Ojo

Para que Kogito pueda serializar y persistir tu objeto necesita un constructor sin argumentos y getters/setters (estilo JavaBean). Es exactamente lo mismo que pide Jackson para convertir desde y hacia JSON. Si te falta el constructor vacío, verás errores de deserialización al iniciar el proceso.

¿Te recuerda a algo? Es el mismo POJO que usarías como cuerpo de una petición REST en Spring. Aquí no escribes ningún @RestController ni @RequestBody a mano: Kogito autogenera el endpoint del proceso y usa este objeto como la forma del JSON que entra y sale.

Declarar la variable en el proceso

Dentro del proceso BPMN declaras una variable a nivel de proceso, le das un nombre (solicitud) y le dices su tipo (SolicitudCredito). En el editor visual de Kogito lo haces desde el panel de propiedades del lienzo; por debajo, el archivo .bpmn guarda algo así:

proceso-credito.bpmn (fragmento)
<!-- Tipo del dato: tu POJO de dominio -->
<bpmn2:itemDefinition id="_solicitudItem"
                      structureRef="com.academy.credito.model.SolicitudCredito"/>

<!-- La variable del proceso, con etiquetas de entrada y salida -->
<bpmn2:property id="solicitud" itemSubjectRef="_solicitudItem">
  <bpmn2:extensionElements>
    <customTags>input</customTags>
    <customTags>output</customTags>
  </bpmn2:extensionElements>
</bpmn2:property>

Esas etiquetas (tags) controlan cómo se comporta la variable frente a la API REST autogenerada. Las que más usarás:

EtiquetaQué hace
inputLa variable se expone como entrada en el POST inicial. El JSON del cuerpo la rellena.
outputLa variable se incluye en la respuesta que devuelve el proceso al completarse.
requiredDebe venir obligatoriamente al iniciar; si falta, la petición se rechaza.
internalVariable de uso interno: se oculta de la API REST (ni entra ni sale).
readonlyNo puede modificarse una vez inicializada.
✅ Tip

Una misma variable puede ser input y output a la vez. Es justo lo que queremos con solicitud: entra con los datos del cliente y sale enriquecida con el scoreCrediticio, el aprobado y el motivoDecision que las tareas fueron escribiendo.

Data input y data output de cada tarea

Las variables viven a nivel del proceso, pero las tareas no operan directamente sobre ellas: cada tarea tiene sus propios parámetros locales. El puente entre ambos mundos es el mapeo de datos:

  • Data input (entrada): copia el valor de una variable del proceso hacia un parámetro de la tarea antes de ejecutarla. Es lo que la tarea lee.
  • Data output (salida): copia el resultado de la tarea de vuelta a una variable del proceso después de completarse. Es lo que la tarea escribe.

En el archivo .bpmn el mapeo se ve como asociaciones de origen y destino. No tienes que escribirlo a mano (lo configura el editor), pero conviene reconocerlo:

Mapeo en el .bpmn (fragmento)
<!-- ENTRADA: la variable del proceso -> parámetro de la tarea -->
<dataInputAssociation>
  <sourceRef>solicitud</sourceRef>        <!-- variable del proceso -->
  <targetRef>_solicitudIn</targetRef>      <!-- parámetro local de la tarea -->
</dataInputAssociation>

<!-- SALIDA: resultado de la tarea -> de vuelta a la variable -->
<dataOutputAssociation>
  <sourceRef>_solicitudOut</sourceRef>     <!-- resultado de la tarea -->
  <targetRef>solicitud</targetRef>        <!-- vuelve a la variable del proceso -->
</dataOutputAssociation>

Y la implementación de esa tarea de servicio es simplemente un método de un bean de Spring. Kogito le inyecta lo mapeado como entrada y guarda lo que regresa como salida:

ScoreService.java
@Component
public class ScoreService {

    // La tarea de servicio invoca este método.
    // Recibe 'solicitud' (data input) y devuelve la versión enriquecida (data output).
    public SolicitudCredito calcularScore(SolicitudCredito s) {
        int score = 700;
        if (s.isTieneMora())            score -= 250;
        if (s.getEdad() < 21)            score -= 50;
        if (s.getIngresosMensuales()
              .compareTo(s.getMontoSolicitado()) > 0) score += 80;

        s.setScoreCrediticio(score);
        return s; // el motor mapea el retorno de vuelta a la variable 'solicitud'
    }
}
ℹ️ Nota

Como pasamos el mismo objeto de entrada y lo devolvemos modificado, el cambio en scoreCrediticio se refleja en la variable solicitud del proceso. Las siguientes tareas ya verán ese valor actualizado.

Del JSON del POST a las variables… y de vuelta

Recuerda del tema anterior: el endpoint del proceso se autogenera. No escribes el controlador. El nombre del proceso (su id) define la ruta, y las variables etiquetadas con input definen la forma del JSON que recibe.

1) El POST inicial llena las variables

Al hacer POST /solicitudesCredito con un cuerpo JSON, Kogito deserializa ese JSON sobre las variables de entrada del proceso. Aquí, el JSON rellena la variable solicitud:

POST /solicitudesCredito
# El cuerpo se mapea sobre la variable de entrada "solicitud"
{
  "solicitud": {
    "nombre": "Ana Hernández",
    "edad": 34,
    "ingresosMensuales": 28000,
    "montoSolicitado": 90000,
    "plazoMeses": 24,
    "tieneMora": false
  }
}

2) El proceso corre y las tareas escriben sobre el estado

Mientras la instancia avanza, las tareas leen solicitud, calculan y escriben de vuelta: el scoreCrediticio, la bandera aprobado, el motivoDecision. Todo dentro de la misma variable.

3) El resultado final se devuelve

Cuando la instancia llega al evento de fin, Kogito serializa las variables etiquetadas con output y las devuelve en la respuesta. Verás algo así:

201 Created — respuesta
{
  "id": "a7f3c8e1-2b54-4d9a-9c11-6e0d2f8b4a30",
  "solicitud": {
    "nombre": "Ana Hernández",
    "edad": 34,
    "ingresosMensuales": 28000,
    "montoSolicitado": 90000,
    "plazoMeses": 24,
    "tieneMora": false,
    "scoreCrediticio": 780,   # lo escribió la tarea de score
    "aprobado": true,           # lo escribió la tarea de decisión
    "motivoDecision": "Score suficiente y sin mora"
  }
}
⚠️ Ojo

Si una variable no lleva la etiqueta output, no aparecerá en la respuesta aunque tenga valor. Y si una variable es internal, ni entra por el JSON ni sale en la respuesta: es de uso exclusivamente interno del flujo.

El id que regresa es el identificador de la instancia del proceso, como el Long id autogenerado de una entidad JPA cuando guardas con save(). Con ese id puedes consultar la instancia, completar tareas humanas o abortarla, igual que recuperarías una entidad por su clave con findById().

El recorrido completo de solicitud

Veamos el panorama completo: la variable solicitud entra por el evento de inicio y va siendo leída y escrita por las tareas a lo largo de todo el flujo, hasta salir por el evento de fin.

  1. Inicio. El POST llega; el JSON llena solicitud con los datos del cliente.
  2. Validar datos. Lee solicitud, comprueba reglas básicas (edad mínima, monto positivo) y puede escribir un motivoDecision si algo falla.
  3. Calcular score. Lee solicitud y escribe scoreCrediticio.
  4. Decidir crédito. Lee el scoreCrediticio y tieneMora, y escribe aprobado y motivoDecision.
  5. Fin. Las variables output se serializan y vuelven en la respuesta.

Tabla de variables del proceso de crédito

En este proyecto trabajamos con una sola variable de proceso, solicitud, que es de tipo objeto y agrupa todos los datos. Para que veas el detalle, desglosamos también los campos que viajan dentro de ella y en qué dirección fluyen respecto a la API REST:

NombreTipoDirecciónDescripción
solicitud SolicitudCredito entrada salida Variable única del proceso. Entra con los datos del cliente y sale enriquecida con el resultado de la evaluación.
solicitud.nombre String entrada Nombre del solicitante.
solicitud.edad int entrada Edad en años; se usa en la validación y en el score.
solicitud.ingresosMensuales BigDecimal entrada Ingreso mensual declarado; clave para evaluar capacidad de pago.
solicitud.montoSolicitado BigDecimal entrada Cantidad de crédito que se pide.
solicitud.plazoMeses int entrada Plazo en meses para pagar el crédito.
solicitud.tieneMora boolean entrada Indica si el solicitante registra mora previa; penaliza el score.
solicitud.scoreCrediticio int salida Lo escribe la tarea Calcular score; base para la decisión.
solicitud.aprobado Boolean salida Resultado de la decisión: aprobado o rechazado.
solicitud.motivoDecision String salida Explicación legible de por qué se aprobó o rechazó.
✅ Tip

Mantén juntos en un mismo objeto los datos de entrada y los de salida cuando representan al mismo concepto del negocio. Así una sola variable solicitud con etiquetas input y output cubre todo el ciclo: menos mapeos, menos ruido y un modelo más fiel al dominio.

Para recordar

Estado

Las variables del proceso son el estado compartido de la instancia, vivo del inicio al fin y persistido por el motor.

Mapeo

Cada tarea lee con data input y escribe con data output. El motor copia los valores por ti.

REST

Las etiquetas input / output deciden qué entra en el POST y qué vuelve en la respuesta.

⛔ Importante

Error común al empezar: olvidar la etiqueta output en la variable y luego sorprenderse de que la respuesta del proceso venga vacía. Si calculaste un scoreCrediticio pero no lo ves al terminar, revisa primero las etiquetas de la variable y el mapeo de salida de la tarea.
