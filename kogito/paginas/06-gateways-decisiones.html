Tema 06

Gateways: bifurcar y unir el flujo

Un proceso casi nunca es una línea recta. Aquí aprenderás a tomar decisiones, lanzar caminos en paralelo y volverlos a juntar — sin escribir un solo if a mano ni tocar un Thread.

¿Qué es un gateway?

Hasta ahora tu proceso ha sido una secuencia: empieza, ejecuta una tarea, ejecuta otra y termina. Pero la vida real tiene condiciones (si el monto supera cierto límite, pídelo a un humano) y tiene concurrencia (mientras consultas al buró de crédito, validas la identidad). El elemento de BPMN que gobierna esos cruces de caminos es el gateway (compuerta o pasarela).

Un gateway es un punto de control del flujo: ahí el camino se bifurca (un flujo entra y salen varios) o se vuelve a unir (varios flujos entran y sale uno). Por sí mismo no ejecuta lógica de negocio ni invoca servicios; solo decide por dónde y cuántos caminos continúan. Se dibuja siempre como un rombo, y una marca interior (X, +, O…) indica de qué tipo es.

Gateway = rombo Evento = círculo Tarea = rectángulo redondeado Flujo de secuencia = flecha

Piénsalo como el plano de control de tu lógica. En tu servicio de Spring decides el camino con if/else y, cuando quieres concurrencia, lanzas un CompletableFuture o un ExecutorService. Los gateways son exactamente eso, pero declarados en el diagrama: el motor de Kogito los interpreta y orquesta por ti. Tu código solo aporta los datos sobre los que se decide.

ℹ️ Nota

Un gateway puede actuar como divergente (split: un flujo entra, varios salen) o convergente (join: varios entran, uno sale). El mismo tipo de rombo se usa para ambos roles; lo determina cuántas flechas lleguen y cuántas partan.

Los tres tipos que usarás todo el tiempo

BPMN define varios gateways, pero el 95% de tus procesos se resuelve con tres. Vamos uno por uno, cada uno con su marca, su diagrama y su equivalente en código que ya conoces.

1) Gateway exclusivo (XOR · marca X)

Toma un solo camino. Evalúa las condiciones de sus flujos de salida en orden y se va por la primera que sea verdadera; el resto se ignora aunque también fueran verdaderas. Es, literalmente, tu cadena if / else if / else.

2) Gateway paralelo (AND · marca +)

Activa todas las ramas a la vez. No evalúa condiciones: cuando diverge (fork), lanza cada camino simultáneamente; cuando converge (join), espera a que todas las ramas entrantes lleguen antes de continuar. Es tu «lanza tres tareas en paralelo y reúnelas todas al final».

3) Gateway inclusivo (OR · marca O)

Es el punto medio: evalúa las condiciones de salida y activa todas las que sean verdaderas — pueden ser una, varias o todas. Cuando converge, espera únicamente a las ramas que realmente se activaron. Útil cuando varios caminos no son mutuamente excluyentes (un cliente puede requerir aviso por correo y por SMS, o solo uno, o ninguno).

✅ Tip

Regla mental rápida: X = uno, + = todos, O = los que cumplan. Si dudas entre exclusivo e inclusivo, pregúntate: «¿los caminos se excluyen entre sí?». Si sí, usa exclusivo (XOR). Si pueden coexistir, inclusivo (OR).

Condiciones en los flujos de salida

En los gateways exclusivo e inclusivo, lo que decide hacia dónde va el flujo no es el rombo, sino las condiciones que escribes en cada flecha de salida. Esas condiciones son expresiones booleanas sobre las variables del proceso (las que viste en el tema anterior). El paralelo no usa condiciones: siempre dispara todo.

En Kogito el dialecto más común para estas condiciones es Java. Una condición es una expresión que devuelve true o false usando las variables del proceso por su nombre:

Condiciones de los flujos (dialecto Java)
// Flujo hacia "APROBADO": las variables del proceso se usan por su nombre
score >= 700 && monto <= 200000

// Flujo hacia "REVISION"
score >= 600 && score < 700

// Flujo "por defecto": sin expresión. Es tu rama else.

Internamente, el editor de procesos guarda estas condiciones en el archivo .bpmn como expresiones formales. Si abres el XML verás algo así para cada flujo condicionado:

credito.bpmn
<sequenceFlow id="flujoAprobado" sourceRef="gwDecision" targetRef="tareaAprobar">
  <conditionExpression xsi:type="tFormalExpression"
                       language="http://www.java.com/java">
    score >= 700 && monto <= 200000
  </conditionExpression>
</sequenceFlow>

Lo importante: tú no escribes este XML a mano, igual que en Spring no escribes el bytecode. Lo dibujas en el editor de procesos y tecleas la condición en la propiedad del flujo. Y, como en Spring, el endpoint REST que dispara el proceso se autogenera: no hay un @RestController que mantengas tú. Mandas un POST con el JSON de entrada y Kogito instancia el proceso y deja que los gateways hagan su trabajo.

ℹ️ Nota

En el gateway exclusivo el orden de evaluación importa: gana la primera condición verdadera. Por eso conviene ordenar de la más específica a la más general y dejar siempre una rama «por defecto» al final, tal como pondrías el else al cierre de tu cadena de if.

El ejemplo del crédito, paso a paso

Retomemos el proceso de solicitud de crédito. Tras calcular un score y comparar el monto, una tarea de reglas deja en la variable decision uno de tres valores: APROBADO REVISION RECHAZADO. Un gateway exclusivo ramifica el flujo según ese valor.

Las tres condiciones de salida, sobre la variable decision (un String), quedarían así:

Condiciones del gateway de crédito
// Flujo → Desembolsar
"APROBADO".equals(decision)

// Flujo → Revisión
"REVISION".equals(decision)

// Flujo → Notificar rechazo  ➜  marcado como "Por defecto", sin expresión

Y para arrancar el proceso, recuerda: el endpoint se autogenera. Solo mandas el JSON de entrada y el motor resuelve el gateway por ti.

arrancar-credito.http
POST http://localhost:8080/credito
Content-Type: application/json

{
  "solicitante": "Ana López",
  "monto": 150000,
  "score": 720
}
// El motor evalúa las reglas, fija decision="APROBADO"
// y el gateway exclusivo dirige el flujo a "Desembolsar".
✅ Tip

Cuando una rama del exclusivo se decide por un valor de texto ("APROBADO", "REVISION"…), prefiere una tarea de reglas que clasifique y deje el veredicto en una variable. El gateway queda limpio y legible, y la lógica de negocio vive en las reglas — el mismo principio de separar el «qué decidir» del «cómo enrutar».

Errores clásicos: fork sin join y casos sin cubrir

⚠️ Ojo

Todo fork paralelo necesita su join. Si abres tres ramas con un gateway paralelo y no las vuelves a unir con otro gateway paralelo, el proceso puede quedarse esperando para siempre por un camino que ya nadie va a cerrar — o, peor, terminar de forma inconsistente con ramas a medias. La regla práctica: cada + que divide tiene un + que reúne.

⛔ Importante

Cuida que las condiciones cubran todos los casos. En un gateway exclusivo, si ninguna condición se cumple y no definiste una rama «por defecto», el motor lanza una excepción en tiempo de ejecución porque no sabe por dónde seguir. Es el equivalente a una cadena de if sin else que se queda sin retorno: siempre define la rama por defecto.

Visualmente, así se ven el patrón correcto (fork con su join) frente al patrón roto (fork sin cierre):

Tabla resumen: los tres gateways

GatewaySímboloSemánticaAnalogía en código
Exclusivo XOR X (rombo con equis) Toma un solo camino: el de la primera condición verdadera. Necesita rama por defecto. if / else if / else — o un switch sobre una variable.
Paralelo AND + (rombo con cruz) Activa todas las ramas a la vez (fork) y, al converger, espera a todas (join). Sin condiciones. Varios CompletableFuture lanzados juntos + allOf(...).join().
Inclusivo OR O (rombo con círculo) Activa todas las ramas cuya condición sea verdadera (una, varias o todas) y espera solo a las activadas. Varios if independientes (no else if): cada uno se evalúa por separado.

El cierre que vale la pena recordar: el exclusivo es tu if/else, declarado en el diagrama; y el paralelo es lanzar varias tareas concurrentes y hacer join, pero sin que tú toques Thread, ExecutorService ni CompletableFuture. El motor de Kogito orquesta la concurrencia, la espera y la unión de ramas. Tú decides la topología del flujo; el motor se encarga de ejecutarla con seguridad.

De un vistazo

  1. Un gateway es un rombo que bifurca o une el flujo; no ejecuta lógica de negocio, solo enruta.
  2. El exclusivo (X) toma un camino según la primera condición verdadera: es tu if/else. Siempre con rama por defecto.
  3. El paralelo (+) lanza todas las ramas a la vez y luego las une esperando a todas: concurrencia sin manejar hilos.
  4. El inclusivo (O) activa todas las ramas cuya condición se cumpla, ni más ni menos.
  5. Las condiciones son expresiones booleanas sobre las variables del proceso; cuídalas para no dejar casos sin cubrir.
  6. Cada fork tiene su join. Y, como siempre, el endpoint que dispara el proceso se autogenera.
