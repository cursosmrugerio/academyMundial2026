Tema 02

De Spring Boot a Kogito

Kogito no es un servidor nuevo que aprender: es una dependencia Maven más dentro del proyecto Spring Boot que ya dominas. Aquí verás qué es, de dónde viene y cómo encaja con todo lo que ya sabes.

¿Qué es Kogito?

Kogito es el motor de automatización de negocio cloud-native de la familia Apache KIE. Te permite ejecutar dos cosas dentro de una aplicación moderna:

  • Procesos descritos en BPMN 2.0 (el flujo de un pedido, una alta de cliente, una solicitud de crédito…).
  • Decisiones y reglas descritas en DMN o Drools/DRL (¿aprobamos este préstamo?, ¿qué descuento aplica?).

El nombre viene del latín cogito («pienso»), con la K de Kubernetes: la pista de que su razón de ser es correr ligero y escalar en contenedores. Y aquí está la buena noticia: Kogito corre sobre Spring Boot o sobre Quarkus. En este curso usamos Spring Boot, así que sigues en casa.

⚙️ Runtime de procesos (BPMN) 🧠 Motor de reglas (DMN / Drools) ☁️ Cloud-native (Kubernetes, nativo) 🌱 Sobre Spring Boot o Quarkus

Piensa en Kogito como piensas en Spring Data JPA: tú no escribes el SQL de cada consulta, declaras una interfaz y el arranque te genera la implementación. En Kogito tú no escribes el @RestController de cada proceso: dibujas el proceso en un archivo .bpmn y el endpoint REST aparece solo, generado durante la compilación.

El salto de jBPM 7.x a Apache KIE 10.x

Si ya habías oído hablar de jBPM, conviene aclarar cómo cambió el mundo. La generación clásica (jBPM 7.x) giraba en torno a un servidor central llamado KIE Server: un monolito al que desplegabas tus procesos empaquetados como artefactos KJAR, y que los interpretaba en tiempo de ejecución a través de una API REST genérica.

Apache KIE 10.x —y Kogito como su runtime— le da la vuelta por completo a esa idea:

  • De monolito a microservicio. En lugar de un servidor gigante que aloja todos los procesos, cada servicio Kogito es tu aplicación, con tus procesos compilados dentro. Escalas el servicio que lo necesita, no el servidor entero.
  • REST específico y autogenerado. Olvídate de la API genérica del KIE Server con su containerId y su jerga. Un proceso orders expone POST /orders, GET /orders/{id}, etc. Endpoints con nombres de tu dominio.
  • Generación de código en compilación, no interpretación. Al compilar con Maven, Kogito traduce tus .bpmn y .dmn a clases Java reales. No hay un intérprete leyendo XML en caliente: hay bytecode. Arranca más rápido, gasta menos memoria y permite incluso compilación nativa con GraalVM.
✅ Tip mental

jBPM 7.x se parecía a desplegar un WAR en un servidor de aplicaciones compartido. Kogito se parece a tu microservicio Spring Boot autocontenido: un java -jar y listo. El motor viaja dentro de tu binario.

ℹ️ Nota

La conexión entre el motor de procesos y el de reglas tiene nombre propio en BPMN: una Business Rule Task. Es el punto donde el flujo se detiene a preguntar «¿qué decido aquí?» y delega en DMN o Drools. Lo verás a detalle más adelante en el curso.

Cómo encaja con tu proyecto Spring Boot

Esta es la parte que de verdad importa: no cambias de tecnología, sumas una capacidad. Un proyecto Kogito sobre Spring Boot es, literalmente, un proyecto Spring Boot normal con tres ingredientes extra:

  1. Una dependencia Maven más. Agregas el starter de Kogito a tu pom.xml, igual que agregaste alguna vez spring-boot-starter-web o spring-boot-starter-data-jpa.
  2. Tus procesos como recursos. Los archivos .bpmn y .dmn viven en src/main/resources, exactamente donde pones tus plantillas, tus .sql o tu application.properties. Son recursos del proyecto.
  3. El endpoint aparece solo. Al compilar, Kogito lee esos recursos y genera el controlador REST. Tú nunca escribes @RestController para arrancar un proceso.
pom.xml
<!-- El starter de Kogito sobre Spring Boot: procesos + reglas -->
<dependency>
  <groupId>org.jbpm</groupId>
  <artifactId>jbpm-with-drools-spring-boot-starter</artifactId>
  <version>10.2.0</version>
</dependency>
Estructura del proyecto
src/main/
├─ java/org/acme/        # tu código Spring de siempre (@Service, @Component…)
└─ resources/
   ├─ orders.bpmn        # el proceso → genera POST /orders, GET /orders/{id}…
   ├─ pricing.dmn        # una decisión → invocable desde el proceso
   └─ application.properties
⚠️ Ojo

El .bpmn es solo XML, pero no lo escribes a mano: lo dibujas con el editor visual BPMN (extensión de VS Code o el editor de la consola). Lo verás en el tema siguiente. Aquí lo importante es entender dónde vive el archivo y qué produce: un endpoint.

El mapa mental: Spring ⟷ Kogito

Casi todo lo que haces en Kogito tiene un gemelo que ya conoces en Spring. No estás aprendiendo conceptos nuevos desde cero: estás cambiando quién escribe el código repetitivo. Antes lo escribías tú con anotaciones; ahora lo deriva el motor a partir del modelo.

En Spring Boot lo hacías con…En Kogito se vuelve…Qué cambia para ti
@RestController + @PostMapping para iniciar algo Endpoint de proceso autogenerado a partir del .bpmn No escribes el controlador: arranca una instancia con POST /orders
@Service con un if/else enorme de lógica de negocio Business Rule Task que invoca DMN / Drools Las reglas salen del Java y se vuelven declarativas y auditables
@Repository / JPA para persistir entidades Persistencia del motor (add-on JDBC sobre PostgreSQL) El estado de cada instancia se guarda solo; no diseñas esas tablas
Un @Scheduled para reintentos o esperas Timer Event / Jobs Service del proceso Las esperas y plazos forman parte del diagrama, no de tu código
ApplicationEventPublisher / colas para integrar Message / Signal Events (CloudEvents sobre Kafka) El proceso reacciona a eventos sin que cablees los listeners
✅ La idea clave

En Spring, tú eres el orquestador: escribes el método que llama a un servicio, luego a otro, manejas el if, guardas en la base. En Kogito, el diagrama es el orquestador y tú aportas las piezas concretas (qué hace cada tarea). El flujo deja de estar escondido entre líneas de Java y se vuelve un mapa que cualquiera puede leer.

Una decisión ya tomada: Spring Boot

Kogito ofrece dos runtimes oficiales y equivalentes en capacidades: Spring Boot y Quarkus. El modelo de procesos, el BPMN, el DMN y la generación de endpoints son idénticos en ambos; cambian sobre todo las dependencias y algún detalle de configuración.

ℹ️ En este curso usamos Spring Boot

Lo elegimos por continuidad: ya conoces su ciclo de vida, su application.properties, su spring-boot:run, sus pruebas con JUnit y Mockito. Quarkus es la otra opción válida —brilla en arranque ultrarrápido y compilación nativa con GraalVM— pero aquí no queremos que aprendas dos cosas a la vez. Todo lo que veas se traslada a Quarkus cambiando los starters.

terminal
# Arrancar tu servicio Kogito: el mismo comando de Spring Boot de siempre
mvn clean compile spring-boot:run

# Iniciar una instancia del proceso "orders" (endpoint autogenerado)
curl -X POST http://localhost:8080/orders \
  -H "Content-Type: application/json" \
  -d '{ "cliente": "ACME", "total": 1500 }'

Fíjate: no hubo paso de despliegue a ningún servidor, ni un @RestController que escribir. Compilaste, arrancaste y el endpoint del proceso ya estaba ahí, esperando. Ese es, en una frase, el espíritu de Kogito.

Lo que te llevas de este tema

  • Kogito es el runtime cloud-native de Apache KIE para procesos (BPMN) y decisiones (DMN/Drools), y corre sobre tu Spring Boot.
  • El salto desde jBPM 7.x es de monolito a microservicio, de API genérica a REST autogenerado, y de interpretar XML a generar código en compilación.
  • Integrarlo es agregar una dependencia Maven y poner tus .bpmn/.dmn como recursos: el endpoint aparece solo.
  • Cada pieza tiene su gemelo en Spring: endpoint de proceso ↔ @RestController, DMN/regla ↔ @Service, persistencia del motor ↔ @Repository/JPA.
